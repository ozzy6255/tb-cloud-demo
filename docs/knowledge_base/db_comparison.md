# 数据库选型: SQL Server vs SQLite

## 知识框架

### SQLite: "数据库在文件中"
- **本质**: 不是服务器，而是一个 C 语言库。它直接读写磁盘上的单一文件 (`app.db`)。
- **优点**: 
    - **零配置**: 无需安装，Python 自带。
    - **单文件**: 备份就是复制粘贴文件。
    - **极快**: 还没等发起网络请求，它已经读完磁盘了。
- **缺点**:
    - **并发弱**: 不适合成百上千人同时**写**操作（会锁文件）。
    - **功能少**: 不支持复杂的存储过程、用户权限管理。

### SQL Server / MySQL / PostgreSQL: "数据库在服务器中"
- **本质**: 独立运行的软件进程，监听端口 (如 1433, 3306)，等待客户端通过网络连接。
- **优点**:
    - **高并发**: 支持巨大的吞吐量。
    - **功能强**: 事务、集群、细粒度权限控制。
- **缺点**:
    - **重**: 即使没人用，它也要吃掉几百兆内存空转。
    - **贵**: 云端托管通常按小时计费。

## 决策树 (Decision Tree)
1.  **数据量** < 100GB 且 **并发** < 100 QPS？ -> **SQLite** (足够 99% 的个人项目使用)
2.  需要 **微信小程序** 或 **多台服务器** 共享数据？ -> **MySQL / Postgres** (云数据库)
3.  遗留系统必须兼容 **T-SQL**？ -> **SQL Server**
